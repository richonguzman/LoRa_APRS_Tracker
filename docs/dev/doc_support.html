map_gps_filter.cpp

lib/MapGPSFilter/map_gps_filter.cpp · map_gps_filter.h

Module de filtrage GPS et de gestion de la trace de position propre. Encapsule toute la logique de qualité GPS — de la donnée brute TinyGPS++ à la position affichable à l'écran. Source unique de vérité pour la position et la trace sur la carte.

Pipeline de filtrage — updateFilteredOwnPosition()

Appelée toutes les ~500ms depuis map_refresh_timer_cb(). Prend un TinyGPSPlus& en entrée, met à jour ownPositionLat/Lon en sortie. Quatre étapes en série :

GPS brut (TinyGPS++) │ ▼ [1] Santé minimale : isValid() + sats >= 3 + sats <= 90 │ KO → return (position inchangée) ▼ [2] Spike rejection : vitesse calculée vs dernière position │ > 150 km/h → return (saut GPS ignoré, log WARN) ▼ [3] Jitter filter : vitesse GPS < 1.5 km/h → return (appareil immobile) │ │ + Poor signal freeze : HDOP > 8.0 ET vitesse < 5 km/h → return ▼ [4] Lissage low-pass : alpha = f(vitesse, HDOP) │ vitesse >= 30 km/h → alpha = 1.0 (GPS direct, pas de lag) │ sinon → alpha = 1/HDOP (HDOP 1 = alpha 1.0, HDOP 5 = alpha 0.2) │ ownPositionLat += alpha * (newLat - ownPositionLat) ownPositionLon += alpha * (newLon - ownPositionLon) │ ▼ Log CSV sur SD : "gps_lat,gps_lon,speed,hdop,alpha,filt_lat,filt_lon,delta_m"
Alpha : plus il est petit, plus la position filtrée est stable (peu réactive aux variations GPS) mais lente à bouger. HDOP=1 (très bon signal) → alpha=1.0, pas de lissage. HDOP=5 (signal moyen) → alpha=0.2, fort lissage.

Gestion de la trace — addOwnTracePoint()

Appelée après chaque updateFilteredOwnPosition(). Enregistre la position filtrée dans le buffer circulaire si :

  1. getUiPosition() retourne true (position valide)
  2. Distance depuis le dernier point > 3m (~0.000027°)

Quand le buffer est plein (500 points) : appel à compactTrace() avant d'ajouter.

Compaction — compactTrace()

Libère de la place dans le buffer en simplifiant les points les plus anciens :

1. Lineariser le buffer circulaire → tableau temporaire linear[500] 2. Appliquer Douglas-Peucker sur la première moitié (points anciens) ε = 0.0002° (~22m) : élimine les points redondants en ligne droite 3. Rebuilder : points gardés de la 1ère moitié + tous les points de la 2ème moitié 4. ownTraceCount = writeIdx 5. ownTraceHead = writeIdx % 500 ← le % est critique (fix bug : writeIdx peut = 500)
Invariant circulaire : ownTraceHead doit toujours être dans [0..499]. Si Douglas-Peucker ne simplifie rien (tous les points gardés), writeIdx peut valoir exactement 500. Sans le % OWN_TRACE_MAX_POINTS, ownTrace[500] serait un accès hors-limite au prochain addOwnTracePoint(). Fix appliqué dans commit 454ffd5.

Tests unitaires

Le module est compilable en dehors d'ESP32 (macro UNIT_TEST). Les dépendances Arduino et ESP-IDF sont mockées. Douglas-Peucker est remplacé par un stub qui garde tous les points. Tests dans lib/MapGPSFilter/test/.

sd_logger.cpp

src/sd_logger.cpp · include/sd_logger.h

Logger persistant sur carte SD. Deux rôles distincts : (1) écriture proactive d'événements système (boot, heartbeat, screen state), (2) capture automatique des logs ESP W/E via hook vprintf. Fournit aussi le mécanisme de crash context en RAM RTC pour diagnostiquer les PANIC sans USB.

Fichier de log

/LoRa_Tracker/system.log       ← fichier actif (max 100KB)
/LoRa_Tracker/system.log.old   ← rotation automatique quand > 100KB

Format d'une ligne

[03-20 08:11:42] INFO  BOOT: Reset reason: POWER_ON
[+000:01:23.456] WARN  MapEngine: No NAV pack for France-Sud Z12

Le timestamp utilise l'heure GPS réelle (MM-DD HH:MM:SS) dès qu'un fix GPS est reçu. Avant le premier fix : uptime relatif (+HHH:MM:SS.mmm).

Mécanisme du hook vprintf

Installé au boot via esp_log_set_vprintf(). Intercepte tous les logs ESP_LOG* avant qu'ils partent vers la serial. Filtre : seuls W (warn) et E (error) sont persistés sur SD. Les niveaux I/D/V sont laissés passer sans écriture SD.

Anti-réentrance : le flag inSdHook empêche la récursion. Si l'écriture SD elle-même génère un ESP_LOGW (ex: SD full), le hook détecte inSdHook=true et ne re-entre pas. Sans ça : stack overflow garanti.

Mutex SD

sdLogMutex protège les accès SD contre la concurrence entre le hook vprintf (appelé depuis n'importe quelle tâche) et les appels explicites à log()/logf(). Timeout : 50ms pour le hook (non bloquant), 1000ms pour les appels explicites.

Crash Context (RAM RTC)

Mécanisme pour diagnostiquer les PANIC sans USB connecté. Fonctionne car la RAM RTC (RTC_NOINIT_ATTR) survit à un reset (PANIC, WDT) mais pas à une coupure d'alimentation.

Pendant l'exécution (toutes les 5s depuis loop(), à chaque render) : updateCrashContext("MODULE", lat, lon) → écrit dans _crashCtx (RAM RTC) : magic = 0xC0FFEE42 module = "LOOP" | "MAP_NAV" | "MAP_RASTER" | "MAP_IDLE" uptimeMs, lat, lon, freeHeap, freePsram, loopCount++ Au boot suivant (dans logBootInfo()) : logPreviousCrashContext() → si reset reason = PANIC ou WDT : si magic == 0xC0FFEE42 → log CRIT dans system.log sinon → log WARN "no crash context" → si reset normal → efface magic (= 0)
Module loguéSignification
LOOPMain loop (loop() dans LoRa_APRS_Tracker.cpp)
MAP_NAVRender task Core 0, pendant renderNavViewport()
MAP_RASTERRender task Core 0, pendant renderRasterViewport()
MAP_IDLERender task Core 0, après rendu, en attente
